# `~/.vim/` audit — suggested improvements

A review of the current Vim configuration with concrete, non-breaking improvements.
Grouped by impact so you can pick what's worth doing.

---

## A. Probable bugs (worth fixing first)

| #  | Where                  | Issue |
|----|------------------------|-------|
| A1 | `colorscheme.vim:18`   | The augroup uses `autocmd ColorScheme default ...` but your active colorscheme is `tokyonight`. The augroup never fires. The ~100-line block inside it duplicates the top-level `hi` commands; the duplication only works because the top-level lines execute once at source time. If you ever run `:colorscheme tokyonight` again (e.g., after a plugin reloads it), your customizations disappear. **Fix:** `autocmd ColorScheme *` (or `tokyonight`). Then you can delete the duplicated top-level `hi` block. |
| A2 | `settings.vim:118`     | `autocmd CursorMoved,CursorMovedI * call HighlightWordUnderCursor()` is bare — not wrapped in an `augroup ... autocmd! ... augroup END`. Re-sourcing `.vimrc` keeps adding duplicate autocmds; each `CursorMoved` then runs the function N times. |
| A3 | `coc.vim:72`           | Same issue: `autocmd CursorHold * silent call CocActionAsync('highlight')` is bare. Same duplication risk on re-source. |
| A4 | `settings.vim:65, 85`  | `set ttimeoutlen=10` is set twice. |
| A5 | `settings.vim:9, 52`   | `set nowrap` is set twice. |
| A6 | `~/.vim/.gitignore`    | `doc/tags` (generated by `:helptags`) isn't ignored. Should be — it's a build artifact. |

## B. Folder structure (the biggest pro-vs-amateur tell)

Right now `~/.vim/` is a mix of:

- **Vim's runtime dirs** that *must* live here: `autoload/`, `after/`, `plugged/`, `UltiSnips/`, `snippets/`, `spell/`, `doc/`
- **Your config files** scattered next to them: `settings.vim`, `mappings.vim`, `plugins.vim`, plus 10+ per-plugin files

**Recommendation:** move your own files into `~/.vim/config/` with a `plugins/` subdir for per-plugin configuration:

```
~/.vim/
├── .vimrc                  ← thin orchestrator
├── config/
│   ├── settings.vim
│   ├── mappings.vim
│   ├── commands.vim
│   ├── functions.vim       ← extracted helpers
│   ├── colorscheme.vim
│   └── plugins/
│       ├── _manager.vim    ← Plug 'foo/bar' lines (was plugins.vim)
│       ├── coc.vim
│       ├── fzf.vim
│       ├── nerdtree.vim
│       ├── gitgutter.vim
│       ├── hexokinase.vim
│       ├── quickscope.vim
│       ├── vim-move.vim
│       ├── vim-dadbod.vim
│       └── vimwiki.vim
├── after/                  ← stays
├── autoload/               ← stays (plug.vim lives here)
├── doc/                    ← cleanup: see C3
├── snippets/               ← stays
├── UltiSnips/              ← stays
├── spell/                  ← stays
└── legacy/
    └── text-dimmer.vim     ← move old backup here so it's obviously inactive
```

And `.vimrc` becomes a glob-based loader (load order is enforced by explicit ordering for the bits that matter, glob for the rest):

```vim
" Plugin manager and plugin list — must run first
source ~/.vim/config/plugins/_manager.vim

" Core settings before everything else
source ~/.vim/config/settings.vim
source ~/.vim/config/functions.vim
source ~/.vim/config/colorscheme.vim

" Per-plugin configuration
for s:f in glob('~/.vim/config/plugins/*.vim', 0, 1)
    if s:f !~# '_manager\.vim$'
        execute 'source' s:f
    endif
endfor

" Mappings/commands last so they can reference everything above
source ~/.vim/config/commands.vim
source ~/.vim/config/mappings.vim

if filereadable(expand('~/dadbod_databases.vim'))
    source ~/dadbod_databases.vim
endif
```

This is roughly how tpope, junegunn, and most polished setups are structured.

## C. Per-file cleanup

### C1. `settings.vim` — separate concerns

Two non-trivial helpers are buried at the bottom: `s:OSC52Yank` (lines 34–47) and `HighlightWordUnderCursor` (lines 88–116). Pulling them into `config/functions.vim` lets `settings.vim` be exactly what its name promises — option-setting only.

### C2. `mappings.vim` — extract functions

Same pattern: `SearchAndShow` (60–94), `ExecutePythonLine`/`ExecutePythonVisual` (115–126), `RunTestUnderCursor` (133–157), `RunPytestUnderCursor` (164–188). All belong in `config/functions.vim`. After extraction, `mappings.vim` shrinks to ~80 lines of pure `nnoremap`/`vnoremap` declarations.

Bonus: `RunTestUnderCursor` and `RunPytestUnderCursor` are 95% the same — you could parameterize one helper that takes the runner command.

Also: lines 191–193 are commented-out dead `pytest` mappings. Delete or uncomment.

### C3. `doc/` is misused

`doc/g.txt` and `doc/assembly.txt` look like personal notes. Vim's `doc/` is reserved for plugin help (the `*tag*` syntax). Move these to `~/.vim/notes/` or anywhere outside `doc/`. Then `doc/tags` is purely a build artifact (and gitignored — see A6).

### C4. `commands.vim` is a one-liner

It contains a single `command! Todos`. Either inline it into `mappings.vim` (since the corresponding mapping is there) or merge it into a per-plugin file (it's an fzf wrapper). One-line files are noise.

### C5. `fzf.vim` — DRY the Rg variants

You have 5 nearly identical `command! ... Rg*` blocks (regular Rg, Rgi, Rgic, Rgir, Rgr). The only thing varying is the rg flags. Factor those into a single function that takes a flag string. Saves ~40 lines.

The `s:LogDebug` machinery writing to `/tmp/vim_debug.log` is also live in production — if you don't need it, removing it saves 30 lines.

### C6. `colorscheme.vim` — collapse the duplication

After fixing A1, the augroup block can be the single source of truth — the top-level `hi` commands become unnecessary. The file shrinks to ~150 lines. The redundancy is currently load-bearing only because the augroup pattern is wrong; once the pattern matches, top-level becomes redundant.

### C7. Filename consistency

You have `vim-move.vim`, `vim-dadbod.vim` (kebab with prefix) alongside `fzf.vim`, `nerdtree.vim`, `vimwiki.vim` (no prefix). Normalize to **no prefix** — the file lives in `plugins/` so the namespace is implicit. So `vim-move.vim` → `move.vim`, `vim-dadbod.vim` → `dadbod.vim`.

### C8. `text-dimmer.vim` backup

Since you decided to keep it as a backup, move it to `~/.vim/legacy/text-dimmer.vim` so it's unambiguously inactive. Future you (or anyone reading the repo) will know not to touch it.

## E. Optional nice-to-haves

- **`plug#begin('~/.vim/plugged')` with explicit path** — currently you let vim-plug pick the default. Being explicit makes the repo portable and grep-able.
- **`if has('vim_starting')` guard** around the vim-plug bootstrap — avoids running the `curl` block if `.vimrc` is re-sourced.
- **`:CocInstall` reminder** — your `coc.vim` lists `g:coc_global_extensions` (auto-installed) but a comment noting "these auto-install on first launch" helps newcomers.

## Suggested execution order

If you act on all of this, do it in this order so each step is independently verifiable:

1. **A1–A6** — real bugs and hygiene. ~10 minutes, low risk.
2. **C3, C8, D1, D2, D3** — move files, gitignore, README. ~10 minutes, zero behavior change.
3. **B (folder reorg)** — bigger commit. Move files in one step; update `.vimrc` to source the new paths; verify Vim still starts cleanly.
4. **C1, C2, C4** — function extraction. Pure refactor; verify by simply using Vim normally.
5. **C5, C6** — DRY rg + collapse colorscheme. Independent of everything else.
